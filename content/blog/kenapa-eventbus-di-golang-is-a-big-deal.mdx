## Chapter 1: The Background

Ini terjadi kepada saya beberapa bulan lalu…

Tepatnya pada bulan Juni 2025 lalu, saya baru saja mendapatkan posisi magang sebagai backend engineer di salah satu startup di Jawa Timur. Tidak lama setelah proses onboarding, saya di-assign kedalam project backend berbasis Go dengan arsitektur <a href="https://www.dicoding.com/blog/mengenal-arsitektur-monolith-kelebihan-serta-batasannya/"
target="_blank"
rel="noopener noreferrer" className="cursor-pointer underline">monolith</a>.

Tentunya ini hal yang wajar bukan? Hingga saat ini arsitektur monolith masih sangat relevan digunakan dan kenapa tidak? monolith memudahkan code management, cognitive overhead, proses debugging dan lain-lain(saya bisa jabarkan lebih banyak kelebihan monolith).

## Chapter 2: The Issue

Pada fase awal pengembangan, monolithic jelas punya banyak kelebihan. Namun kompleksitas sistem akan turut meningkat seiring waktu. Seperti:

- Dependencies hell atau ketergantungan dependencies tidak akan terbendung.
- Reliability, artinya apabila salah satu module mengalami eror, maka hal ini akan memberikan efek kepada seluruh sistem.

Dengan adanya isu-isu diatas, maka dibutuhkan adanya suatu mekanisme yang bisa membantu module berkomunikasi tanpa saling menggantungkan satu sama lain(_tight coupling_).

## Chapter 3: The Solution

Salah satu pattern yang bisa dipakai untuk mengurangi _tight coupling_ adalah event-driven communication dengan publish-subscribe. Dengan ini, tiap komponen ataupun service dalam suatu sistem dapat mem-publish/broadcast event sehingga komponen atau service yang lain dapat melakukan subscribe terhadap event tersebut dan meresponnya berdasarkan konten/payload dari masing-masing event.

Pastinya banyak readers yang sudah cukup familiar dengan konsep ini, karena hal ini cukup serupa dengan message queues pada RabbitMQ dan tentunya fitur Pub/Sub pada Redis.

Konsep publish-subscribe ini dapat di achieve dengan fitur concurrency dan channel yang terdapat dalam Go. Sehingga, pada artikel kali ini, saya akan memberikan contoh implementasi publish-subscribe pada aplikasi backend terkait toko online sederhana.

**Hold Up..**

Sebelum masuk ke bagian implementasi, mungkin ada pertanyaan yang mencuat seperti.

> Kenapa tidak langsung saja menggunakan EDA tools seperti Apache Kafka ataupun RabbitMQ?

Karena konsep EDA(Event-Driven Architecture) pada Apache Kafka maupun RabbitMQ diperuntukkan untuk aplikasi dengan event streams yang tinggi, distributed system(seperti microservices) hingga data persistant. Hal ini tidak cocok untuk skala aplikasi low-medium, yang mana merupakan skala aplikasi yang akan saya demokan untuk artikel ini.

Disamping itu, ini terkait dengan simplicity, terutama bagi readers yang belum familiar dengan konsep EDA pada microservices dan hanya sekadar ingin mencicipi performa EDA pada aplikasi monolith dengan penggunaan channels & goroutines sederhana.

## Chapter 4: The Implementation

Pada pembahasan terkait implementasi kali ini, terlebih dahulu saya akan menjabarkan terkait 3 fungsionalitas utama pada eventbus, yakni:

- **Publish** → Fungsi yang melakukan broadcast/publishing event ke seluruh sistem.
- **Subscribe** → Fungsi yang memungkinkan service atau komponen untuk melakukan subscribe kepada event spesifik yang diperlukan.
- **Unsubscribe** → Kebalikan dari fungsi ke-2, ini digunakan untuk melakukan unsubscribe event spesifik yang dirasa telah tidak diperlukan lagi.

**Eventbus Structure & Generics**

Untuk kebutuhan scaffolding eventbus, dibutuhkan sebuah struktur dan beberapa fungsi generics yang nantinya akan mempermudah workflow.

```go
type Event struct {
 Payload any
}
```

`Event` adalah sebuah definisi dari struktur event yang didalamnya terdapat `Payload` sebagai representasi kontekstual dari data sebuah event dengan tipe `any` .

```go
type (
 EventChan chan Event
)
```

`EventChan` adalah alias yang digunakan untuk mengirimkan `Event` dalam bentuk channel.

```go
type EventBus struct {
 mu          sync.RWMutex
 subscribers map[string][]EventChan
}
```

`EventBus` merupakan definisi struktur utama dari eventbus yang didalamnya terdapat 2 buah properti, yaitu:

- `mu sync.RWMutex` membuat proses baca dapat tersinkronasi. Sehingga tidak terjadi tabrakan antar proses baca dan tulis pada `subscribers` .
- `subscribers` adalah sebuah slice yang berisi `map` . Dimana `string` adalah representasi topik/nama event, dan `[]EventChan` adalah slice dengan tipe `EventChan` . `subscribers` digunakan untuk menyimpan seluruh subscribers dengan masing-masing topik event yang berbeda dan tiap subscribers menerima event dari `EventChan` .

Lalu, dibawah ini adalah beberapa fungsi _generics_ yang implementasinya mengikuti konsep dari 3 fungsionalitas utama eventbus diatas.

```go
func (eb *EventBus) Publish(eventName string, event Event) {
 eb.mu.RLock()
 defer eb.mu.RUnlock()

 subscribers := append([]EventChan{}, eb.subscribers[eventName]...)
 go func() {
  for _, subscriber := range subscribers {
   subscriber <- event
  }
 }()
}
```

`Publish()` menerima 2 buah parameter, yaitu:

- `eventName` sesuai namanya, parameter ini akan menjadi subyek dari sebuah event tertentu.
- `event` adalah sebuah event yang memiliki struktur `Event`.

`RLock()` adalah fungsi yang berasal dari `sync.RWMutex` yang digunakan agar beberapa goroutines dapat memanggil `Publish()` secara bersamaan tanpa menganggu value pada `subscribers`. Dibawahnya terdapat `RUnlock()` yang merupakan fungsi yang wajib dipanggil setelah `RLock()`, karena akan merilis lock sesaat setelah `Publish()` berakhir(baik itu error ataupun tidak).

Lalu, `subscribers` adalah hasil copy dari `eb.subscribers` original yang berisi slice dari `EventChan` untuk `eventName` sebagai topiknya.

> _Untuk apa proses copy slice dilakukan?_

Hal ini dilakukan untuk menghindari terjadinya _race conditions_, karena apabila copy tidak dilakukan, maka value dari `subscribers` akan berisi reference pada original slice.

Terakhir, `go func(){...}()` bertujuan untuk membuat beberapa goroutines sesuai dengan jumlah `subscribers`, agar setiap kali `Publish()` dipanggil, proses tidak terkena _block_ akibat tunggu-menunggu. Dengan kata lain, setiap pemanggilan `Publish()`, response dapat segera diberikan sedangkan `subscribers` yang lain dapat tetap berjalan di background.

```go
func (eb *EventBus) Subscribe(eventName string) EventChan {
 eb.mu.Lock()
 defer eb.mu.Unlock()

 ch := make(EventChan)
 eb.subscribers[eventName] = append(eb.subscribers[eventName], ch)
 return ch
}
```

`Subscribe()` menerima `eventName` sebagai satu-satunya parameter yang merupakan subyek dari event yang ditargetkan.

Didalamnya, `Lock()` yang berasal dari `sync.RWMutex` berfungsi untuk memastikan operasi _write_/tulis pada `subscribers` tereksekusi secara bergantian sehingga tidak menimbulkan adanya _overlapping_ pada goroutines. Sama halnya seperti `RUnlock()` yang berlaku untuk `RLock()`, `Unlock()` adalah fungsi yang wajib dipanggil namun berlaku untuk `Lock()`.

Selanjutnya, `ch` merupakan slice dari `EventChan` yang di assign kedalam `subscribers` dengan `eventName` yang relevan.

```go
func (eb *EventBus) Unsubscribe(eventName string, ch EventChan) {
 eb.mu.Lock()
 defer eb.mu.Unlock()

 if subscribers, ok := eb.subscribers[eventName]; ok {
  for i, subscriber := range subscribers {
   if ch == subscriber {
    eb.subscribers[eventName] = append(subscribers[:i], subscribers[i+1:]...)
    close(ch)

    for range ch {
    }
    return
   }
  }
 }
}
```

`Unsubscribe()` menerima 2 buah parameter, yaitu:

- `eventName` yakni subyek event yang menjadi target untuk dilakukannya _unsubscribing_.
- `ch` adalah event dalam bentuk channel yang akan dihapus.

Penggunaan kembali `Lock()` disini dikarenakan `Unsubscribe()` merupakan operasi _write_/tulis sama seperti `Subscribe()` sebelumnya.

Lalu, proses penghapusan event terjadi setelah _conditional check_ lolos secara keseluruhan, maka value `subscribers` akan dimodifikasi dengan mengikutsertakan seluruh event kecuali event `i` , dan pemanggilan `close()` dilakukan untuk mengakhiri lifecycle event.

Pada bagian akhir fungsi, perulangan kosong `for range ch{}` dilakukan untuk memastikan tidak adanya event yang tersisa pada channel buffer.

### **Eventbus in Action**

📢 _Disclaimer_: Project yang saya demokan pada artikel kali ini menggunakan [layered architecture](https://dev.to/yasmine_ddec94f4d4/understanding-the-layered-architecture-pattern-a-comprehensive-guide-1e2j), sehingga apabila terdapat kebingungan dengan navigasi file, silahkan check repository project ini [disini](https://github.com/rayhan889/ecovera-backend).

_Flow_ dari implementasi eventbus ini dimulai dari _trigger_ `Subscribe()` terlebih dahulu, dan disusul oleh `Publish()` .

> _Mengapa demikian? Bukankah secara logika, event perlu di publish terlebih dahulu untuk mendapatkan audience/subscribers?_

Seperti ini analoginya,

- Publish adalah sebuah stasiun radio yang melakukan _broadcast_ sinyal.
- Subscribe adalah masyarakat yang menyalakan radio dan melakukan tuning channel radio.

Apabila tidak ada masyarakat yang menyalakan radio, _broadcast_ sinyal akan menjadi sia-sia.

`app/subscribers/subscriber_manager.go`

```go
package subscribers

import (
 "github.com/rayhan889/ecovera-backend/app/services"
 "github.com/rayhan889/ecovera-backend/pkg/eventbus"
)

// list services yang dibutuhkan oleh subscribers
type SubscriberDependencies struct {
 CartService          *services.CartService
 PaymentDetailService *services.PaymentDetailService
 OrderService         *services.OrderService
 UserService          *services.UserService
 MailService          *services.MailService
}

// interface untuk menentukan basic method yang akan diadopsi oleh
// tiap subscribers
type EventSubscriber interface {
 Subscribe(eventBus *eventbus.EventBus)
}

// struktur utama subscriber manager
type SubscriberManager struct {
 dependencies *SubscriberDependencies
 subscribers  []EventSubscriber
}

// constructor untuk SubscriberManager
func NewSubscriberManger(
  deps *SubscriberDependencies,
  eventBus *eventbus.EventBus,
) *SubscriberManager {
 manager := &SubscriberManager{
  dependencies: deps,
  // agar simpel, constructor dari tiap subscriber untuk event spesifik
  // akan langsung di assign
  subscribers: []EventSubscriber{
   NewOrderSubscriber(deps),
  },
 }

 // assign seluruh subscribers dengan eventBus
 for _, s := range manager.subscribers {
  s.Subscribe(eventBus)
 }

 return manager
}

// fungsi helper untuk mendapatkan list event yang terdaftar
func (sm *SubscriberManager) GetRegisteredEventsName() []string {
 eventsName := make([]string, 0)
 for _, subs := range sm.subscribers {
  eventsName = append(eventsName, subs.GetEventName())
 }

 return eventsName
}

// fungsi helper untuk mendapatkan total subscribers
func (sm *SubscriberManager) GetSubscriberCount() int {
 return len(sm.subscribers)
}
```

_Tasks_ yang terdaftar berdasarkan event `order.created` akan terseksekusi secara berurutan. _Readers_ dapat bebas melakukan modifikasi perihal task apa yang akan ter-_trigger_ berdasarkan event tertentu.

Selanjutnya, sonstructor `NewSubscriberManager()` akan dipanggil di `app.go` sebelum endpoints didaftarkan pada aplikasi. Ini bertujuan agar ketika salah satu endpoint dijalankan, seluruh `subscribers` hanya perlu menunggu event yang sesuai dengan `eventName` masing-masing.

### _Why is this better?_

Pertama, dengan _flow_ seperti ini, seluruh `subscribers` yang sebelumnya telah terdaftar diawal akan menunggu event yang dipublish tanpa mengetahui sumbernya(dalam hal ini order service), begitupun sebaliknya.

Kedua, hal ini membuat baik order service(sebagai _publisher_) dan `subscribers` menjadi terpisah dan dapat tereksekusi secara independen.

Ketiga, tidak adanya P2P(_point-to-point_) integrations, yang mana memudahkan developer apabila kedepannya dirasa perlu menambahkan subscriber baru.

Terakhir, karena `subscribers` telah terinisialisasi sedari awal, `subscribers` dapat segera merespon event yang dipublish secara cepat.

## Chapter 6: The Conclusion

Implementasi EventBus dengan pendekatan Event-Driven di Golang membuktikan bahwa kita bisa mendapatkan modularitas, skalabilitas, dan kecepatan respon yang lebih baik tanpa harus buru-buru pakai solusi _heavyweight_ seperti Kafka atau RabbitMQ. Terutama untuk aplikasi skala low-medium. Dengan fitur dasar seperti `Publish`, `Subscribe`, dan `Unsubscribe` yang didukung goroutines, channels, dan mekanisme locking yang tepat, kode dapat menjadi _loosely coupled_ dan lebih mudah dilakukan kedepannya.

Namun perlu diingat, pendekatan ini memiliki batasan. Apabila aplikasi _readers_ nantinya melibatkan ribuan event per detik, distribusi antar server, atau butuh persistensi event jangka panjang, _readers_ mungkin akan butuh sistem yang lebih kompleks.

Intinya, eventBus(dalam konteks monolith/Golang) adalah senjata yang _powerful_ untuk memperkecil _coupling_(ketergantungan), mempercepat waktu respon, dan menjaga _codebase_ tetap _manageable_, asalkan diimplementasikan dengan awareness terhadap skala dan kebutuhan sistem ke depan.
